# AUDIT_2026-07 — event-ops-v1 read-only codebase audit

**Date:** 2026-07-07 · **Method:** static read-only audit; no files modified, no DB access, no migrations run. Migration files (0001–0029) were treated as schema ground truth — findings that depend on live schema are tagged **[verify live schema]** because CLAUDE.md documents hand-applied SQL as a normal practice.
**Scope note:** issues already documented *and fixed* per CLAUDE.md Known Issues are not re-reported; several findings below are **new instances of those same documented patterns** and are flagged as such.

**Severity:** P0 = breaks pilot or leaks data · P1 = will bite within one pilot · P2 = later.

## 1. RLS write safety

**Core pattern: clean.** No new instances of the documented `createServerClient()`-write-against-a-policy-less-table pattern. All writes to the four SELECT-only tables (`operators`, `operator_users`, `audit_log`, `usage_events`) go through `createAdminClient()`. All RLS-client writes target tables with `FOR ALL` policies (0009/0013/0014/0015/0026/0027/0028). The fixed instance (`src/app/admin/settings/actions.ts`) correctly uses admin client + `.select()` + zero-rows guard.

However, four findings are the same phenotype — a write that silently affects zero rows while the UI/flow reports success — via schema mismatch or missing RLS instead of a missing policy:

- **1.1 · P0** — `supabase/migrations/0025_whatsapp_session_state.sql` — `whatsapp_session_state` has RLS **disabled entirely** ("only accessed via createAdminClient()"), which in Supabase means the table — customer phone numbers and their raw messages (`original_message`) — is readable/writable by anyone holding the public anon key via PostgREST. Fix: `ALTER TABLE whatsapp_session_state ENABLE ROW LEVEL SECURITY;` with no policies (service-role access unaffected). Known pattern: no (inverse — missing RLS, not missing policy).
- **1.2 · P0** — `src/lib/data-entry/change-events.ts:127,163-170` — `propagateToKB` selects/updates `kb_sections.version` and `updated_at`, columns that do not exist per migrations 0004/0018, so the select errors (42703) and `continue`s for every section — **KB propagation after a confirmed mastersheet or WhatsApp change is a total silent no-op** and both callers (`src/app/api/data-entry/confirm/route.ts:243`, `src/lib/data-entry/pending-changes.ts:487`) report success with `kb_sections_updated: []`. Fix: drop `version`/`updated_at` from the select+update (or add the columns) and log/propagate the fetch error instead of `continue`. Known pattern: **yes** — silent zero-row write, new vector. **[verify live schema]**
- **1.3 · P1** — `src/lib/crm/campaigns.ts:225-233` (also `:169-172`) — final campaign status writes `'send_failed'`/`'partial'`/`'failed'`, all rejected by the 0027 CHECK (`draft|sending|sent|paused|cancelled`), and the error is unchecked — campaigns stick at `'sending'` forever while the API tells the user the send completed. Fix: map to allowed statuses (or migrate the CHECK) and check `error` + `.select('id')` zero-rows guard. Known pattern: yes — false-success write.
- **1.4 · P1** — `src/lib/demo/seed-demo-event.ts:240` — demo KB insert omits `kb_sections.kb_document_id` (NOT NULL, 0004:21), fails 23502 unchecked → demo event "seeds successfully" with **zero KB sections**, so the demo simulator can answer nothing. Fix: create a stub `kb_documents` row first and throw on insert `error`. Known pattern: yes — false-success write. **[verify live schema]**
- **1.5 · P2** — Missing post-write `.select()` zero-rows guards on user-facing mutations (all show success on a silent no-op): `src/app/admin/events/[eventId]/escalations/actions.ts:113-123,154-165,202-214` (claim/resolve/reopen check `error` but not row count), `src/app/admin/events/[eventId]/conversations/[conversationId]/actions.ts:78-81` (conversation close entirely unchecked), `src/app/admin/events/new/actions.ts:100` + `src/app/api/events/route.ts:156` (`config.event_id` patch — on failure it stays `''` forever), `src/lib/recovery/payment-recovery.ts:160-167` (`markRecoveryCompleted` — recovery-fee stats depend on it), `src/lib/crm/campaigns.ts:95-98` (racy read-modify-write of `total_recipients`, unchecked). Fix: apply the settings-actions pattern (`.select('id')` + length check) to each. Known pattern: partial — the exact silent-zero-rows mechanism, but policies exist.

Table→policy map confirmed from migrations: everything RLS-on with `FOR ALL` via `current_user_operator_ids()` except `operators`/`operator_users`/`audit_log`/`usage_events` (SELECT-only, admin-written by design) and `whatsapp_session_state` (RLS off — see 1.1). `refund_cases` has zero code writes anywhere (see §11).

## 2. Exact-match fragility

- **2.1 · P1** — `src/lib/whatsapp/adapter-factory.ts:10-18` — the documented production bug is **still unfixed in code**: `provider === 'meta'` / `'360dialog'` compares untrimmed `process.env.WHATSAPP_PROVIDER`, so a re-introduced trailing newline reproduces the "200 but zero outgoing requests" failure today. Fix: `process.env.WHATSAPP_PROVIDER?.trim().toLowerCase()`. Known pattern: **yes — the original instance, fixed operationally only**.
- **2.2 · P1** — `src/app/api/cron/expire-pending/route.ts:11`, `src/app/api/cron/weekly-digest/route.ts:28` — `authHeader !== \`Bearer ${process.env.CRON_SECRET}\``: if `CRON_SECRET` is ever unset the expected value is the literal `"Bearer undefined"`, an auth bypass; a trailing newline instead fails closed (cron 401s forever). Fix: return 500 when unset, trim, use `timingSafeEqual`. Known pattern: yes (newline half); `Bearer undefined` is a new subclass.
- **2.3 · P2** — `src/app/api/whatsapp/inbound/route.ts:58` — GET verify-token compared untrimmed; a newline in `META_WEBHOOK_VERIFY_TOKEN` makes webhook verification 403 forever. Fix: trim both sides. Known pattern: yes.
- **2.4 · P2** — `src/lib/whatsapp/meta-adapter.ts:25-29` and `src/lib/whatsapp/360dialog-adapter.ts:27-31` — both `requireEnv` helpers return untrimmed values: a newline in `META_PERMANENT_TOKEN` corrupts the Authorization header (fetch throws *before* the prized `postToMeta` log), and a newline in `META_APP_SECRET`/`DIALOG360_API_KEY` makes HMAC verification reject every webhook. Fix: `return val.trim()` with an empty-after-trim guard. Known pattern: yes.
- **2.5 · P2** — `src/lib/email/send.ts:19-20,29,65` — `RESEND_API_KEY`/`SENDGRID_API_KEY` interpolated untrimmed into Bearer headers; the truthiness gate also lets a whitespace-only key select a provider that then always fails. Fix: `?.trim() || undefined`. Known pattern: yes.
- **2.6 · P2** — `src/lib/data-entry/dato-connector.ts:34-35` — `DATOCMS_API_TOKEN` untrimmed into the Authorization header. Fix: trim on read. Known pattern: yes.
- **2.7 · P2** — `src/lib/agent/anthropic-client.ts:29` — module-level `new Anthropic({ apiKey: process.env.ANTHROPIC_API_KEY })`: a missing key crashes every importing route at module load with a confusing stack; a newline-suffixed key yields opaque 401s. Fix: trim + explicit missing-var error. Known pattern: yes.
- **2.8 · P2** — `src/lib/supabase/admin.ts:12` — `process.env.SUPABASE_SERVICE_ROLE_KEY!` non-null assertion passes `undefined` into `createClient` on unset var, surfacing as a cryptic downstream error. Fix: explicit guard with a descriptive throw. Known pattern: no (missing-env variant).

Verified clean: both adapters' signature checks already use `timingSafeEqual` with length guards; Meta payload enum comparisons are correct exact matches; `whatsapp/test` correctly validates the untrimmed token against Meta (so it *detects* the newline scenario).

## 3. Auth & IDOR

**Clean overall — no cross-tenant data leak or unauthenticated mutation path found.** All 26 API routes and 7 server-action files perform `auth.getUser()` plus an ownership/membership check before returning or mutating tenant data. The active-operator cookie (`src/lib/get-active-operator.ts:19-23`) is validated against fresh `operator_users` membership on every request and is not spoofable. `src/middleware.ts` uses `getUser()` (not `getSession()`) and correctly scopes `/admin/*`; every `/api/*` handler self-authenticates. Auth callback is clean.

Residual findings:

- **3.3 · P2** — `src/app/api/whatsapp/test/route.ts:22-31` — auth-gated but no operator-membership check; any authenticated user can probe whether Meta credentials are valid (no send, no data). Fix: add a membership check for symmetry. Known pattern: no.

## 4. Date-range & exactly-one-row queries

Uniqueness ground truth from migrations: `promoters UNIQUE(operator_id, phone_e164)` — **not** phone alone; `operators.whatsapp_business_phone_number_id`, `conversations.wa_message_id`, `pending_changes.confirmation_wamid` — **no constraint**; `mastersheet_mappings(operator_id, format_fingerprint)` — non-unique index.

- **4.1 · P1** — `src/app/api/whatsapp/inbound/route.ts:468-473` — promoter lookup filters `phone_e164 + is_active` only (`.maybeSingle()`, error discarded): the same phone whitelisted by two operators errors → null → the promoter **silently falls into the customer-support flow**; even with one row, a promoter of operator A texting operator B's number enters A's change-management flow. Fix: scope the lookup by the operator resolved from `phoneNumberId` and log the error. Known pattern: **yes** — phone-number uniqueness pattern, new instance on `promoters.phone_e164`.
- **4.2 · P1** — `src/lib/agent/whatsapp-router.ts:36-42` — `getOperatorByPhoneNumberId` never reads `error`: a duplicate phone-number-ID (the exact documented incident) now produces a swallowed multiple-rows error and a misleading "No operator found" log — the code still cannot detect recurrence of the known bug. Fix: destructure and log `error`, distinguishing "not found" from "duplicate/failed". Known pattern: **yes** — the exact CLAUDE.md pattern; data was fixed, detection was not.
- **4.3 · P1** — `src/app/api/whatsapp/inbound/route.ts:544-552` + `src/lib/data-entry/pending-changes.ts:70-109` — `createPendingChange` supersedes all pending rows *before* inserting with `inbound_wamid UNIQUE`, so a duplicate webhook delivery supersedes the legit pending change and then throws on the unique constraint — **the promoter's Confirm button now resolves to nothing**. Fix: check `inbound_wamid` existence (idempotency guard) before the supersede step. Known pattern: partial — consequence of missing wamid dedup (see 5.2).
- **4.4 · P1** — `src/lib/crm/campaigns.ts:287-335` — `getNoShowSegment` has no "event has ended" bound and explicitly falls back to **all completed orders** when there are no gate scans (lines 324-326) — a re-marketing blast can target every buyer of an event that hasn't happened yet. Fix: require `end_date < today` (event-local) and return empty, not all orders, when scan data is absent. Known pattern: yes — window must be bounded on both ends.
- **4.5 · P1** — `src/lib/agent/whatsapp-router.ts:84-98` — the recently-ended-draft window has both bounds (documented bug fixed) but computes them from the **UTC** calendar date vs event-local `end_date`: from 21:00 UTC to midnight (00:00–03:00 Saudi) the post-event support window is shifted a day, dropping recently-ended events from routing nightly. Fix: compute date strings in the event/operator timezone (`Intl.DateTimeFormat` with the events `timezone` column). Known pattern: yes — routing-bounds pattern, new timezone variant.
- **4.6 · P2** — `src/lib/agent/state-machine.ts:243-253` — `daysUntilEvent` uses UTC-midnight parsing vs server time to pick **refund tiers**, so the wrong refund percentage can be quoted near tier boundaries; inconsistent with the tz-aware equivalent in `generator.ts`. Fix: reuse `buildEventTimingContext`'s tz-aware local-date diff. Known pattern: yes — off-by-timezone class.
- **4.7 · P2** — `src/lib/agent/generator.ts:136-144` — "Event ended N days ago" is computed from `start_date` (set as `event_date_iso` everywhere), so on day 2+ of a multi-day event the agent claims the event already ended. Fix: pass `end_date` into `EventConfig` and use it for the "ended" branch. Known pattern: no.
- **4.8 · P2** — `src/app/api/cron/weekly-digest/route.ts:36-39` — metrics window `[now-7d, now]` disagrees with the printed period label (`weekAgo … now-24h`); Monday 00:00–09:00 activity is included but labelled out-of-period. Fix: snap both bounds to whole UTC days and label from those. Known pattern: yes — window-semantics instance (cosmetic).
- **4.9 · P2** — `src/app/admin/events/[eventId]/conversations/page.tsx:41-46` and `src/app/api/events/[eventId]/conversations/export/route.ts:22-27` — "Today" filter uses server (UTC) midnight; a Riyadh operator's Today view omits 00:00–03:00 local conversations. Fix: compute midnight in the event/operator timezone. Known pattern: yes — off-by-timezone.
- **4.10 · P2** — `src/lib/recovery/payment-recovery.ts:210-225` — expiry only transitions `status='pending'`; once `sent`, an attempt never expires despite the message promising "expires in 24 hours" — stats overcount and customers can be re-messaged against a dead link. Fix: include `'sent'` (and `'opened'`) in the expiry filter. Known pattern: yes — one-sided lifecycle bound.
- **4.11 · P2** — `src/lib/gate/scan-validator.ts:98-104` — order lookup uses `.ilike(...).maybeSingle()` on a case-*sensitively*-unique column with the error discarded: case-colliding order IDs → error → valid ticket **rejected at the gate**; `%`/`_` in a QR payload act as wildcards. Fix: exact case-normalised match, escape/reject wildcard chars, read the error. Known pattern: yes — exactly-one-row on a not-quite-unique filter.
- **4.12 · P2** — `src/app/api/data-entry/upload/route.ts:146-176` — the saved-mapping merge ignores `format_fingerprint`, taking the most-recent mapping of **any** format and overriding colliding column names with boosted confidence (bypassing review at 0.85). - **4.13 · P2** — `src/lib/agent/whatsapp-router.ts:102,126-129` — `limit(5)` silently truncates the multi-event selection list; a 6th active event can never be selected, no log. Fix: raise the limit or log truncation. Known pattern: no.

## 5. Multi-turn WhatsApp state

Happy path verified clean: the event-selection round trip correctly persists and re-injects `original_message` (migration 0029 pattern, `inbound/route.ts:195-257`); `expires_at` is enforced on the only session-state read (`whatsapp-session-state.ts:74`); promoter button confirmations are context-tied via `context_wamid`→`confirmation_wamid` with sender re-verification and expiry/race statuses.

- **5.1 · P1** — `src/lib/agent/whatsapp-conversation.ts:103-108` — the "last 10 messages" history query is `.order('created_at', { ascending: true }).limit(10)` — the **first** 10: past 10 total messages, the agent's context window freezes at the opening turns and never sees recent messages, degrading every long thread. Fix: order descending, limit 10, reverse. Known pattern: yes — lost-context class, new mechanism.
- **5.2 · P1** — `src/app/api/whatsapp/inbound/route.ts` (whole POST path) — no `wa_message_id` dedup for the customer flow: Meta webhook redelivery re-runs classifier+generator (double Anthropic spend), inserts duplicate `messages`, and sends the customer two replies; the promoter flow's only "dedup" is the destructive unique-violation in 4.3. Fix: check the wamid against `messages` (or a dedup table) at the top of `handleInboundMessage` and drop duplicates. Known pattern: yes.
- **5.3 · P1** — `src/lib/recovery/payment-recovery.ts:75-168` + `src/lib/crm/campaigns.ts` — recovery/CRM replies have **no inbound context at all**: the message says "Reply STOP to opt out" but the inbound route has no STOP branch and no lookup of open recovery attempts/campaign recipients by phone, so STOP is answered by the AI (compliance risk) and replies drop into the generic agent; additionally `markRecoveryCompleted` has zero callers, so recovery stats report 0% recovered forever. Fix: add an inbound pre-router (STOP handling + phone match against open attempts/recipients) and wire `markRecoveryCompleted` to a webhook or manual action. Known pattern: yes — persist-then-reinject pattern absent from these new flows.
- **5.4 · P1** — `src/app/api/whatsapp/inbound/route.ts:143-150` — event selection fuzzy-matches on the event name's **first word** (`lowerText.includes(e.name.split(' ')[0])`): a follow-up like "what's the difference between them?" containing a common first token silently selects an event and answers the persisted question against the wrong event. Fix: require the full event name or a high-similarity match, never one leading token. Known pattern: yes — selection round-trip pattern.
- **5.5 · P2** — `src/app/api/whatsapp/inbound/route.ts:218-225` — a *new question* sent while a selection is pending is neither answered nor stored (only the list is re-sent), so it is permanently lost. Fix: update `original_message` with the new text on parse failure. Known pattern: yes — persist-then-reinject violated on this branch.
- **5.6 · P2** — `src/lib/agent/whatsapp-session-state.ts:43-58` + `inbound/route.ts:247-251` — if the session-state write fails (logged only), the numbered prompt is still sent; the customer's "1" then finds no state and is itself persisted as `original_message: "1"` — the exact bug migration 0029 fixed, resurrected via the failure path. Fix: surface the write failure and skip the numbered prompt (degrade to a single-turn answer). Known pattern: yes.
- **5.7 · P2** — expired `whatsapp_session_state` rows are never deleted (`cron/expire-pending` handles only `pending_changes` and `payment_recovery_attempts`); the table grows unboundedly with one-time phones. Fix: add `delete().lt('expires_at', now)` to the cron. Known pattern: no.

### 1.3 status-mapping choice + details-field finding
